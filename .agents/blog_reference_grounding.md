# HUGS Blog Grounding：参考博客的可复用原则

本文件是后续撰写 HUGS 博客大纲、文案与页面实现时的依据。它总结自以下参考页：

- TeleDexter：<https://bigai-dex.github.io/blog/teledexter/>
- GEN-1：<https://generalistai.com/blog/gen-1>

目标不是复制其视觉或措辞，而是复用它们将研究主张转化为可信、可读长文的方式。

## 1. 两个参考页各自提供什么

### TeleDexter：研究型技术叙事

- 从一个读者能立刻理解的能力缺口开篇，而不是从论文方法图或摘要开篇。
- 先提出贯穿全文的中心命题，再逐步解释缺失的能力层、方法如何构建、系统能做什么、数字与限制。
- 文章本身像一篇可连续阅读的研究论文解读：克制、具体、技术上可追溯。
- 版式采用窄正文阅读列、左侧粘性元数据/目录、白底和单一强调色；视觉服务于阅读而不是装饰。
- 每个视频、图表和图注都有明确论证作用，而不是素材陈列。

### GEN-1：能力主张与证据节奏

- 以一个大判断建立文章的意义，再把证据按能力维度组织，而不是机械复述论文目录。
- 每个能力维度遵循“展示 → 量化 → 解释”的顺序：先让读者看到行为，再给出数字和实验语境。
- 视频、对比、图表和长时程演示被视为论证材料；读者应知道每段媒体要观察什么。
- 在结尾保留 limitations，使强主张有清晰边界。
- 视觉系统有很强的品牌性和交互性，但其真正值得借鉴的是信息层级、证据编排和阅读节奏。

## 2. 共同的长文方法

### 先建立问题，再介绍系统

文章开头应依次回答：

1. 什么能力缺口值得解决？
2. 为什么现有路径不足？
3. 本文提出的框架改变了哪个关键环节？

对于 HUGS，这个问题不是单纯“怎样生成一个抓取姿态”，而是如何通过 Human → Sim → Real 路径，让机器人在跨尺度、跨 `contact modes` 的抓取中获得更好的先验、物理知识与真实部署能力。

### 用一条中心命题贯穿全文

HUGS 博客的中心命题应保持一致：Human 提供 object-conditioned prior，Sim 提供可扩展的 physics knowledge，Real 检验可部署性；三者共同服务于统一灵巧抓取。

任何章节、图片或视频都应能回答该命题的一个子问题。不能回答时，不应仅为“丰富页面”而加入。

### 以证据单元而非素材堆叠组织内容

每个关键主张都应形成一个完整单元：

```text
问题或主张
  → 一段直观解释 / 可观看的演示
  → 量化图表或可核验事实
  → 对结果边界的解释
```

例如，讲“多模式在部署中的价值”时，应先给同一物体的不同抓取视频，再解释场景约束，最后引用实际实验能够支持的范围；不能只展示视频或只报一个数字。

## 3. HUGS 应采用的叙事框架

### 文章层级

1. **大框架**：为什么灵巧手操作需要 Human → Sim → Real。
2. **能力定位**：抓取追求尽可能宽的新物体泛化；通用操作突出 contact-rich dexterity。二者互相依赖，不做绝对切分。
3. **形态意义**：灵巧手与双手协作扩展了可处理的物体几何、尺度与接触模式空间；不将平行夹爪描述为“完全不可能”。
4. **具体难题**：跨尺度、跨模式的统一抓取为何形成离散—连续的大搜索空间。
5. **HUGS 的回答**：object-conditioned human prior 如何提出 `contact modes` 与 `wrist initializations`，再由 `force-closure-aware optimization` 落实为机器人抓取。
6. **Human / Sim / Real 证据链**：人类先验、物理合成与真实部署必须相互衔接。
7. **边界与愿景**：限制、未解决问题和可验证的下一步。

### 语气与范围

- 允许有大 scope，但要将“研究判断/未来方向”与“论文已验证结果”分开写。
- 将 Human → Sim → Real 称作“重要路径”或“有前景的路径”，不称作已被证明的唯一最终路径。
- 不使用“首个”“彻底解决”“human-level”“任意物体都能抓”等绝对化结论，除非有明确的论文定义与证据。
- 不把 HUGS 表述为已经解决通用操作；它是统一 dexterous grasp synthesis 的工作。

## 4. 页面与阅读体验

### 推荐版式

- 桌面端：左侧粘性栏承载主页入口、元数据和目录；中央标题卡；窄正文列承载长文。
- 移动端：标题 → 元数据 → 主视觉 → 目录 → 正文；侧栏不固定，内容保持单列。
- 正文应比项目主页更窄、更连续，减少并排卡片；复杂信息用图、短段落和小标题分层。
- 使用大留白、细分割线、克制的强调色和高可读性图注。延续 HUGS 现有视觉亲缘关系，但不复制参考页的字形、配色或品牌风格。

### 目录与章节

- 长文必须有目录；桌面端可粘性显示，移动端为普通目录卡。
- 目录标题应映射读者问题，而不是只复制论文小节名称。
- 每节只承担一个主要论点；若一节包含多个论点，拆为连续的小节。

### 媒体使用

- 首屏最多使用一个主视觉（teaser 图片或 overview 视频），避免同时竞争注意力。
- 视频默认静音，提供 controls 和 `playsinline`；正文在视频前说明观察重点。
- 图像要尽可能大，以便检视；图注采用“结论 + 必要上下文”，不写成密集论文 caption。
- 图表与视频必须有文本解释；不要只依靠颜色表达类别、对比或结论。
- 优先使用已有 HUGS 资产；不为装饰生成无证据含量的新视觉。

## 5. 事实、术语与实现约束

### 事实表达

- 每一项数字、比较、成功率、规模和真实世界结论均应来自论文或用户明确提供的材料。
- 对相关工作、平行夹爪和 VLA/WAM 的讨论应说明能力边界，避免无证据的贬低或绝对化。
- limitations 必须来自论文或用户提供材料；不能为“显得客观”而编造。

### 术语一致性

始终使用：

- `HUGS`
- `Human-prior-guided`
- `Unified dexterous Grasp Synthesis`
- `object-conditioned human prior`
- `contact modes`
- `wrist initializations`
- `force-closure-aware optimization`
- `multi-mode`（不用 `multi-modal`）

### 静态站实现

- 保持静态 HTML、CSS 和轻量 JavaScript；不引入构建系统。
- 为图片提供有意义的 `alt` 文本；为视频和交互提供可理解的文字说明。
- 保持移动端单列可读；尊重 `prefers-reduced-motion`。
- 不修改现有主页，博客页面与样式保持独立。

## 6. 发布前的最终自检

- 文章是否先解释了“为什么”，再介绍 HUGS？
- 每一个核心主张是否有对应的视觉或量化证据？
- 视频、图表是否各自服务一个明确的论点？
- 大愿景与已验证结果是否明确区分？
- 目录、正文、媒体和移动端是否都可连续阅读？
- 是否所有术语、数字、链接和资源路径都经过核验？
